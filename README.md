## 2. Философия: сначала продукт, потом код

Главная ошибка новичков — сразу открывать терминал и писать `claude "сделай мне приложение"`.

**Правильный порядок:**

```
Идея → Описание продукта → Анализ рынка → MVP-границы → Стек → Критика → Код
```

Claude Code — мощный инструмент, но он работает тем лучше, чем точнее ты знаешь, что хочешь получить. Размытое описание даёт размытый результат.

**Чем сильнее исходное описание — тем полезнее результат от Claude Code.**

## 3. Шаг 1 — Формулируем идею

Прежде чем что-то строить, ответь на три вопроса:

1. **Какую проблему я решаю?** — Не «я хочу сделать приложение», а «люди тратят 2 часа в день на X, я хочу сократить это до 10 минут».
2. **Для кого?** — Конкретный пользователь, не «все». Студенты? Фрилансеры? Команды из 5-10 человек?
3. **Почему это стоит делать?** — Что изменится, если продукт появится? Почему сейчас?

### Формула хорошей идеи

```
[Кто] сейчас страдает от [проблемы], потому что [причина].
Мой продукт [решение], которое позволяет [результат].
```

**Пример:**
> Начинающие разработчики тратят часы на настройку dev-окружения, потому что инструкции разбросаны по десяткам источников. Мой продукт — интерактивный установщик, который автоматически настраивает всё за 15 минут с пошаговой проверкой.

## 4. Шаг 2 — Project Description

Project Description — это главный документ проекта. На нём строится всё остальное.

### Что должно быть в описании

| Раздел | Что включить |
|---|---|
| **Что это** | Тип продукта: веб-приложение, бот, CLI-инструмент, API-сервис |
| **Для кого** | Целевая аудитория с конкретикой |
| **Проблема** | Что именно не работает сейчас |
| **Решение** | Как продукт решает проблему |
| **Сценарий** | Путь пользователя от первого контакта до результата |
| **Отличие** | Чем лучше/проще/быстрее существующих решений |

### Как получить хороший Project Description

Используй Claude Code или Claude в чате — дай ему свою идею и попроси структурировать. Промпт ниже в разделе [Готовые промпты](#13-готовые-промпты).

**Важно:** не принимай первый вариант как финальный. Перечитай, задай уточняющие вопросы, попроси переписать слабые места.

## 5. Шаг 3 — Анализ рынка и конкурентов

Даже учебный проект стоит проверить на конкурентов — это помогает понять, что действительно стоит делать.

### Что выяснить

- Какие похожие продукты/стартапы уже есть
- Их сильные и слабые стороны
- Что они делают хорошо (не нужно изобретать это заново)
- Где у них пробелы (это твоя возможность)

### Как анализировать

1. Попроси Claude найти и сравнить аналоги
2. Посмотри Product Hunt, GitHub Trending, Reddit
3. Сформулируй своё отличие: быстрее, дешевле, проще, точнее, уже (для конкретной ниши)

## 6. Шаг 4 — Roadmap и MVP

### Что такое MVP

MVP (Minimum Viable Product) — это **минимальная версия**, которая уже решает проблему. Не половина продукта, а самая маленькая целая версия.

### Правила MVP

- **Включай** только то, без чего продукт не работает
- **Исключай** всё, что можно добавить позже
- **Если сомневаешься** — выноси за пределы MVP

### Структура Roadmap

```
MVP (неделя 1-2):
├── Функция А — ядро продукта
├── Функция Б — минимальный UI
└── Функция В — базовая интеграция

v1.1 (неделя 3-4):
├── Функция Г — улучшенный UX
└── Функция Д — аналитика

v2.0 (позже):
├── Функция Е — масштабирование
└── Функция Ж — монетизация
```

### Как составить Roadmap

Попроси Claude разбить проект на этапы, но обязательно проверь:
- Не слишком ли жирный MVP?
- Можно ли убрать что-то и продукт всё ещё работает?
- Есть ли чёткая граница между «первая версия» и «потом»?

## 7. Шаг 5 — Технологический стек

### Не просто перечисли — объясни

Плохо:
```
Frontend: React
Backend: Node.js
Database: PostgreSQL
```

Хорошо:
```
Frontend: React + Tailwind CSS
  → React — потому что компонентный подход ускоряет разработку с AI
  → Tailwind — быстрая стилизация без написания CSS вручную

Backend: Node.js + Express
  → Один язык на фронте и бэке — проще для solo-разработчика
  → Express — минимальный фреймворк, не перегружает MVP

Database: SQLite (MVP) → PostgreSQL (v2)
  → SQLite для старта: zero config, один файл, проще деплой
  → Миграция на PostgreSQL когда появится нагрузка
```

### На что обратить внимание при выборе стека

| Критерий | Почему важно |
|---|---|
| Знакомство | Что ты уже знаешь хотя бы базово? |
| AI-совместимость | Насколько хорошо Claude Code работает с этим стеком? (React, Next.js, Python/FastAPI — отлично) |
| Скорость старта | Как быстро можно получить рабочий прототип? |
| Деплой | Куда и как будет выкладываться? (Vercel, Railway, VPS) |
| Зависимости | Чем меньше внешних сервисов в MVP, тем лучше |

## 8. Шаг 6 — Критический разбор

Первый вариант описания, roadmap и стека **почти всегда сырой**. Критический проход — обязательный этап.

### Метод ролей

Попроси Claude (или коллегу) пройтись по проекту с позиции трёх ролей:

| Роль | На что смотрит |
|---|---|
| **CEO** | Есть ли рынок? Будет ли кто-то платить? Зачем это миру? |
| **Product Manager** | Понятен ли пользовательский путь? Не перегружен ли MVP? Приоритеты расставлены? |
| **Tech Lead** | Реализуем ли стек? Есть ли технические риски? Что сломается при масштабировании? |

### Что делать с результатами

1. Прочитай критику
2. Раздели на «справедливо» и «не актуально для MVP»
3. Внеси правки в описание, roadmap и стек
4. **Повтори проход ещё раз** — второй раунд ловит то, что пропустил первый

## 9. Шаг 7 — Настраиваем репозиторий

Теперь, когда проект спроектирован, создаём репозиторий.

### Через терминал

```bash
# Создаём папку проекта
mkdir my-project && cd my-project

# Инициализируем git
git init

# Создаём репозиторий на GitHub
gh repo create my-project --public --source=. --remote=origin

# Первый коммит
echo "# My Project" > README.md
git add README.md
git commit -m "Initial commit"
git push -u origin main
```

### Структура проекта

Рекомендуемая стартовая структура (зависит от стека):

```
my-project/
├── README.md           ← описание проекта
├── CLAUDE.md           ← инструкции для Claude Code
├── .gitignore          ← что не коммитить
├── package.json        ← (для JS/TS проектов)
├── requirements.txt    ← (для Python проектов)
├── src/                ← исходный код
│   └── ...
├── tests/              ← тесты
│   └── ...
└── docs/               ← документация (если нужна)
    ├── project-description.md
    ├── roadmap.md
    └── tech-stack.md
```

### .gitignore

Не забудь сразу добавить `.gitignore`. Claude Code может сгенерировать его:

```bash
claude "создай .gitignore для проекта на [твой стек]"
```

## 10. Шаг 8 — CLAUDE.md и инструкции для агента

`CLAUDE.md` — специальный файл в корне проекта, который Claude Code автоматически читает при старте. Это твой способ дать агенту контекст без повторения одних и тех же инструкций.

### Что писать в CLAUDE.md

```markdown
# Project: My Project

## Описание
[Краткое описание проекта — 2-3 предложения]

## Стек
- Frontend: React + Tailwind
- Backend: FastAPI
- Database: SQLite
- Деплой: Vercel (фронт) + Railway (бэк)

## Структура
- src/frontend/ — React-приложение
- src/backend/ — FastAPI сервер
- tests/ — тесты (pytest)

## Команды
- `npm run dev` — запуск фронта
- `uvicorn main:app --reload` — запуск бэка
- `pytest` — тесты

## Правила
- Пиши код на русском (комментарии) и английском (код)
- Используй TypeScript, не JavaScript
- Тесты обязательны для бизнес-логики
- Не добавляй зависимости без явного запроса
```

### Зачем это нужно

- Claude Code **не помнит** предыдущие сессии (если не настроена память)
- Каждый раз, когда ты запускаешь `claude`, он начинает с нуля
- `CLAUDE.md` — это способ дать ему нужный контекст моментально

## 11. Шаг 9 — Начинаем кодить с Claude Code

### Принцип: от скелета к деталям

Не проси Claude сразу написать всё приложение. Иди поэтапно:

```
1. Скелет проекта (структура файлов, конфиги, точка входа)
2. Базовая функциональность (одна главная фича)
3. Проверка: запускается? Работает?
4. Следующая фича
5. Повторяем 3-4
```

### Хорошие промпты для старта разработки

```bash
# Создать скелет
claude "Создай структуру проекта по CLAUDE.md. Только файлы и базовая настройка, без бизнес-логики."

# Первая фича
claude "Реализуй [конкретную функцию]. Она должна [что делать]. Входные данные: [X]. Результат: [Y]."

# Проверить и исправить
claude "Запусти проект и проверь, что [конкретный сценарий] работает. Если есть ошибки — исправь."
```

### Чего НЕ делать

| Плохо | Хорошо |
|---|---|
| «Сделай мне приложение» | «Создай REST API для регистрации пользователей с JWT-авторизацией» |
| «Исправь баги» | «При отправке формы на /register возвращается 500. Посмотри логи и исправь» |
| «Сделай красиво» | «Добавь Tailwind-стили к компоненту Header: тёмный фон, белый текст, фиксированная позиция» |
| «Добавь всё из roadmap» | «Реализуй функцию поиска из MVP: поле ввода + фильтрация по названию» |

## 12. Шаг 10 — Итерации и доработка

### Цикл разработки

```
Написал фичу → Проверил → Работает? → Да → Коммит → Следующая фича
                                      → Нет → Дебаг → Исправил → Проверил
```

### Коммиты

Делай коммиты часто и с понятными сообщениями:

```bash
# Через Claude Code
claude "закоммить текущие изменения"

# Или вручную
git add .
git commit -m "feat: add user registration endpoint"
git push
```

### Когда MVP готов

- [ ] Все фичи из MVP-списка реализованы
- [ ] Приложение запускается без ошибок
- [ ] Основной пользовательский сценарий работает от начала до конца
- [ ] Код в репозитории на GitHub
- [ ] README описывает, как запустить проект

### Промпт 1. Старт проекта — Project Description

```text
Помоги мне спроектировать продукт.

Идея:
[опиши свою идею в 3-5 предложениях]

Сделай:
1. Полный Project Description:
   — что это за продукт
   — для кого
   — какую проблему решает
   — как работает
   — какой у него основной сценарий использования

2. Анализ рынка:
   — какие есть конкуренты
   — какие стартапы и похожие решения уже существуют
   — их сильные и слабые стороны

3. Предложи улучшения:
   — чем можно выделиться
   — какие уникальные функции добавить
   — как сделать продукт более понятным и конкурентным
```

```text
На основе описания проекта сделай:

1. Roadmap:
   — что входит в MVP (минимум для рабочего продукта)
   — что входит в следующую версию
   — какие этапы реализации логичны

2. Технологический стек:
   — фронтенд
   — бэкенд
   — база данных
   — интеграции
   — AI-инструменты (если применимо)

3. Объясни, почему выбран именно этот стек и где у него ограничения.

Учитывай, что проект будет разрабатываться одним человеком с помощью Claude Code.
```

### Промпт 3. Критический разбор

```text
Проанализируй мой проект с разных ролей:

— CEO: есть ли рынок? зачем это миру?
— Product Manager: понятен ли путь пользователя? не перегружен ли MVP?
— Tech Lead: реализуем ли стек? что сломается при росте?

Сделай:
1. Найди слабые места проекта
2. Покажи риски
3. Укажи, что не продумано
4. Предложи конкретные улучшения
5. Скажи, что убрать, упростить или перенести на потом

Будь максимально критичным.
После этого предложи улучшенную версию Project Description, Roadmap и стека.
```

### Промпт 4. Инициализация кода (новый)

```text
Я готов начать разработку. Контекст проекта — в CLAUDE.md.

Сделай:
1. Инициализируй проект: package.json / requirements.txt / go.mod (что подходит)
2. Создай структуру папок
3. Настрой линтер и форматтер
4. Добавь .gitignore
5. Создай точку входа (пустую, но рабочую — приложение должно запуститься)
6. Добавь скрипты для запуска в dev-режиме

```

```text

1. Проверь, что проект запускается

```

## 14. Частые ошибки

### На этапе проектирования

| Ошибка | Как избежать |
|---|---|
| **Слишком абстрактная идея** — по описанию нельзя понять, что собирать | Используй формулу: кто + проблема + решение + результат |
| **Нет границ MVP** — в первую версию попало всё | Правило: если убрать фичу и продукт всё ещё работает — убирай |
| **Стек без обоснования** — «React потому что все используют» | Каждый выбор = ответ на вопрос «почему именно это?» |
| **Пропущен критический проход** — первый вариант принят как финальный | Минимум один проход по ролям (CEO/PM/Tech Lead) |
| **Смешение описания и кода** — сразу лезут в реализацию | Сначала полный цикл проектирования, потом код |

### На этапе разработки

| Ошибка | Как избежать |
|---|---|
| **«Сделай всё»** — один огромный промпт на весь проект | Разбивай на маленькие задачи: одна фича = один промпт |
| **Нет CLAUDE.md** — каждый раз объясняешь контекст заново | Создай CLAUDE.md в первый день |
| **Не коммитят** — весь код в рабочей директории | Коммить после каждой рабочей фичи |
| **Не проверяют** — доверяют, что Claude написал всё правильно | Всегда запускай и проверяй после каждого изменения |
| **Не читают код** — не понимают, что сгенерировано | Читай и разбирай. Если не понимаешь — спроси Claude объяснить |

## 17. Cartographer — карта проекта для экономии токенов

### Проблема

Каждый раз, когда ты начинаешь новую сессию Claude Code или даёшь новую задачу в большом проекте, агент тратит значительное количество токенов на изучение структуры проекта: читает файлы, ищет зависимости, разбирается в архитектуре. В крупных проектах это может съедать **тысячи токенов** ещё до того, как начнётся реальная работа.

### Решение

**[Cartographer](https://github.com/kingbootoshi/cartographer)** — плагин, который создаёт полную карту кодовой базы в файле `docs/CODEBASE_MAP.md`. При следующем запуске Claude Code читает эту карту вместо того, чтобы заново обходить весь проект.

**Результат:** старт новой задачи тратит значительно меньше токенов, потому что агент сразу понимает структуру проекта, зависимости между модулями и потоки данных.

### Как это работает

```
Без Cartographer:                    С Cartographer:
                                     
Новая задача                         Новая задача
  → Читает 20-50 файлов               → Читает CODEBASE_MAP.md (1 файл)
  → Ищет зависимости                   → Сразу знает структуру
  → Разбирается в архитектуре          → Идёт к нужным файлам
  → Только потом начинает работу       → Начинает работу
  
  ~5000-15000 токенов на вход          ~1000-3000 токенов на вход
```

### Установка

```bash
# 1. Установить зависимость
pip install tiktoken
# или
uv pip install tiktoken

# 2. Добавить плагин
# В интерактивном режиме Claude Code:
/plugin marketplace add kingbootoshi/cartographer
/plugin install cartographer

# 3. Перезагрузить Claude Code
```

### Использование

```bash
# В интерактивном режиме Claude Code:
/cartographer

# Или просто напиши:
claude "map this codebase"
```

После запуска:

1. Cartographer запускает параллельные AI-агенты для анализа проекта
2. Создаёт файл `docs/CODEBASE_MAP.md` с полной картой: архитектура, файлы, зависимости, потоки данных
3. Добавляет ссылку на карту в `CLAUDE.md`, чтобы агент подхватывал её автоматически

### Когда обновлять карту

- После добавления новых модулей или значительного рефакторинга
- Повторный запуск `/cartographer` обновляет **только изменённые модули** — не тратит токены на пересканирование всего проекта

### Когда использовать

| Размер проекта | Нужен Cartographer? |
|---|---|
| Маленький (до 10-15 файлов) | Нет — Claude и так быстро разберётся |
| Средний (20-50 файлов) | Рекомендуется — экономия станет заметной |
| Большой (50+ файлов) | Обязательно — без карты каждая новая задача дорогая |
